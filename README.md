# 基于 py 的 Claude 代理脚本

## 需求背景

### 1. 问题描述

用户在本地安装了一个 Claude CLI，但没有订阅 Claude 服务，且所处的网络环境不支持访问 Anthropic 的 API。

但是，用户可以通过 ssh 连接到一个远程服务器（fdse@10.176.37.2），这台远程服务器上储存有可用的 Claude CLI 的 OAuth 令牌，且其所在的网络环境支持访问 Anthropic 的 API。

### 2. 解决思路

设计一个 py 脚本，当用户在本地终端输入 fdseClaude 命令时，脚本会下载远程服务器上的 OAuth 令牌到本地，并建立网络代理，将本地 Claude CLI 的请求转发到远程服务器上，再由远程服务器将请求转发到 Anthropic 的 API 上进行处理，最后将结果返回给本地用户。

### 3. 预期目标

用户在本地输入 fdseClaude 命令后，能够使用本地的 Claude CLI 进行稳定地交互。

### 4. 注意事项

1. 不要把需求错误的理解为使用远程的 Claude CLI，远程服务器的作用仅为存储 OAuth 令牌和转发请求。
2. 用户的操作系统可能是windows10/11，macos，Linux你要做好跨平台处理。
3. 每次你执行拉取和推送令牌你都应该遵守拉取条件，推送条件。
  - 3.1 拉取条件：运行时本地令牌失效且远端令牌有效（检测本地 OAuth 令牌是否有效，通过 Python 的 subprocess 在本地执行 `fdseClaude -p "hi"`命令进行试探。若输出中不含401则说明令牌有效，反之则说明为失效。检测远程服务器OAuth 令牌，通过 Python 的 subprocess 执行 ssh 命令（如 `ssh fdse@10.176.37.2 "claude -p "hi"`）在远程端进行同样的试探。若输出中不含401则说明令牌有效，反之则说明为失效。）
  - 3.2 推送条件：运行时本地令牌变化(令牌守护任务)且本地令牌有效（有效性检测同上）
## 启动时逻辑

本脚本在启动时同时启动两个进程：

第一进程：
### 1. 代理路径探测和建立

当输入 fdseClaude 命令时，开始探测远程代理服务器的可用性，并在节点不可用时尝试自动重建：

1. 检查端口 port = 8080 是否可连接
   - 若可连接，进一步检测是否为 htps 进程
     - 若是，进入下一步。
     - 若不是，提示用户占用 8080 端口的进程信息，并询问用户是否要尝试在 port + 1 启动 htps 进程
       - 若用户同意，尝试在 port + 1 端口启动 htps 进程
         - 若启动成功，进入下一步。
         - 若启动失败，最多重试 5 个端口，每次重试无需询问用户。若仍失败，提示用户无法启动 htps 进程，终止脚本。
       - 若用户不同意，终止脚本。
   - 若不可连接，则自动尝试在 8080 端口启动 htps
     - 若启动成功，进入下一步。
     - 若启动失败，最多重试 3 次，若仍失败，提示用户无法启动 htps 进程，终止脚本。
2. 检查 1080 端口是否可连接
   - 若可连接，进一步检测是否为 ssh 进程
     - 若是，进入下一步。
     - 若不是，提示用户占用 1080 端口的进程信息，终止脚本。
   - 若不可连接，提示用户 1080 端口无法连接，终止脚本。
3. 检查 ssh 进程是否存在
   - 若存在，检查连接的远程服务器是否为 fdse@10.176.37.2，若有多个 ssh 连接，只要有一个连接到 fdse@10.176.37.2 即可 
     - 若是，则进一步检查与 fdse@10.176.37.2 的 ssh 连接是否正常
     （连接正常是指本地主机能正常与远程主机通信无需检测远程主机的网络环境是否正常）
       - 若正常，宣告代理路径建立成功。
       - 若不正常，则杀死这个僵尸 ssh 进程，尝试重新建立一个 ssh 连接到 fdse@10.176.37.2
         - 若重新建立成功，宣告代理路径建立成功。
         - 若重新建立失败，最多重试 3 次，若仍失败，提示用户无法建立 ssh 连接，终止脚本。
     - 若不是，则尝试建立一个 ssh 连接到 fdse@10.176.37.2
       - 若建立成功，宣告代理路径建立成功。
       - 若建立失败，最多重试 3 次，若仍失败，提示用户无法建立 ssh 连接，终止脚本。
   - 若不存在，则尝试建立一个 ssh 连接到 fdse@10.176.37.2
     - 若建立成功，宣告代理路径建立成功。
     - 若建立失败，最多重试 3 次，若仍失败，提示用户无法建立 ssh 连接，终止脚本。
   > 注意：建立 ssh 连接时要使用持久连接，因为 Claude CLI 可能会执行长耗时的 bash 命令，这时不会发送请求，ssh 可能会因为没有请求而断开连接，所以需要使用持久连接来保持 ssh 连接的稳定性。 

第二进程：

### 2. 启动时 OAuth 令牌管理

#### 2.1 git仓库检查

先检测.claude文件夹有没有初始化git
- 若有进入下一步
- 若无，则尝试帮用户初始化要求如下：该git仓库仅追踪下的.credentials.json文件
  - 若初始化失败，则报错退出。
  - 若初始化成功，则下一步。

#### 2.2 本地令牌有效性检查

  检测本地 OAuth 令牌是否有效，通过 Python 的 subprocess 在本地执行 `fdseClaude -p "hi"`命令进行试探。若输出中不含401则说明令牌有效，反之则说明为失效。
  - 若本地OAuth 令牌有效，则将OAuth 令牌推送至远程服务器，只需推送.claude文件夹下的.credentials.json 文件即可，使用 scp 命令从本地推送至远程服务器指定目录
    - 若推送失败，最多重试 3 次，若仍失败，提示用户无法推送 OAuth 令牌，终止脚本。
    - 若推送成功，则跳过2.3进入下一步

#### 2.3 远端令牌有效性检查

  - 检测远程服务器OAuth 令牌，通过 Python 的 subprocess 执行 ssh 命令（如 `ssh fdse@10.176.37.2 "claude -p "hi"`）在远程端进行同样的试探。若输出中不含401则说明令牌有效，反之则说明为失效。
    - 若远程服务器OAuth 令牌有效，自动从远程服务器上下载 OAuth 令牌到本地，并将其存储在本地 Claude CLI 的配置目录下。只需要下载 .credential 文件即可，使用 scp 命令从远程服务器下载文件到本地指定目录。
      - 若下载失败，最多重试 3 次，若仍失败，提示用户无法下载 OAuth 令牌，终止脚本。
      - 若下载成功，则做一次git commit -m "OAuth token updated by remote at <timestamp>"
    - 若远程服务器OAuth 令牌失效，报错退出。
  
若未报错退出此时此进程结束回到第一进程作为主进程


### 3. 设置终端代理

设置终端的代理为 port，使得本地的 Claude CLI 的请求能够通过代理转发到远程服务器上，设置环境变量 HTTP_PROXY 和 HTTPS_PROXY，值为 http://localhost:port。该设置仅对此对话有效。

### 4. 启动 Claude

在终端中启动 Claude CLI，注意要一并传入参数。

## 运行时逻辑

### 1. 令牌守护任务

本地的 CLaude CLI 可能会刷新 OAuth 令牌，需要监测本地 Claude CLI 的配置目录下的 .credentials.json 文件的变化，当文件发生变化时，自动将新的 OAuth 令牌上传到远程服务器上，保持远程服务器上的令牌与本地的一致。（请告诉我你是如何检测文件变化的）

- 若发生了变化，执行一次git commit -m "OAuth token updated by local CLI at <timestamp> by <useridentifier>"这个useridentifier是用户标识符与用户的电脑绑定且始终不变，至于怎么实现你给出一个合适的方案

### 2. 代理路径守护任务

由于网络不稳定等原因，代理路径，特别是 ssh 节点可能会断开，需要定期检测整条代理路径的可用性，当检测到代理路径不可用时，自动尝试重新建立代理路径，直到成功为止。

### 3. 用户干预任务守护任务

Claude CLI 在以下情况下会中断工作，此时需要用户干预：

- Claude 要求用户对使用的工具审批
- Claude 要求用户回答问题
- Claude 完成了任务并停下
- Claude 因为各种故障停下，如网络故障、令牌过期等

> 注意：在 auto 模式下，Claude 一般不会要求用户对使用的工具审批，请注意区分。

你需要持续监测这些需要用户干预的情况干预的情况，并有以下两种应对模式：

#### 3.1 系统通知模式

该模式致力于解决用户可能无法持续监测终端窗口的情况，因为有时用户会让 Claude 在后台工作，这时如果 Claude 停下了，用户可能无法第一时间发现，系统通知可以提醒用户及时处理。

**启用方法**：用户可以通过在输入 fdseClaude 命令时增加 --notify 参数来启用系统通知模式，这时脚本会在 $USERPROFILE/.claude 下创建 .fdseClaudeNotify 文件来标记系统通知模式启用。该功能默认关闭，这意味着每次正常或非正常退出后都会删除 .fdseClaudeNotify 文件来重置系统通知模式。

> TIP: 使用文件的原因是用户可以随时在对话期间启用或禁用系统通知模式，而不需要重启脚本。

**功能要求**：

当满足以下条件时，需要以系统通知 Toast 的形式通知用户：

1. Claude 因为上述 4 种情况停止了工作。
2. 当前终端和其父窗口失去焦点。

点击通知会将用户的焦点切换到当前终端窗口，方便用户及时处理。

自查清单：

- [ ] 系统通知模式是否仅在窗口失去焦点时触发
- [ ] 若是 IDE 内置终端，是否仅在 IDE 窗口失去焦点时触发，其他父窗口也是如此
- [ ] 是否区分了 auto 模式下的工具审批，例如 auto 模式下 Claude 想要使用 edit 不应该触发通知

## 退出时逻辑

当用户正常退出 Claude CLI 时，或非正常退出时（如强制关闭 Powershell 窗口），需要执行清理工作：

0. 检测本地 Claude CLI 的配置目录下的 .credentials.json 文件的变化
- 若发生变化，则执行2.2 本地令牌有效性检查
- 若未发生变化，继续下一步
1. 清理本地的代理设置，使用 setx 命令将 HTTP_PROXY 和 HTTPS_PROXY 环境变量删除。
2. 清理运行时的守护进程。
3. 保留 ssh、htps 进程，无需关闭，便于复用。
